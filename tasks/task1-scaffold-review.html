Task 1 — Scaffold Monorepo mini-spec approval gate

First code task after Checkpoint 0. This reviews the mini-spec tasks/specs/04-scaffold.md — no code exists yet. On approval: feature branch → TDD/incremental implementation → review/simplify → no-mistakes PR. Size L (scaffold exception: mostly generated code).

Objective

Runnable, gate-ready skeletons for all four top-level parts — frontend/, backend/, config/, devtools/oidc/ — with the hexagonal layout and the anti-god-class Domain structure locked in from the first commit. No feature logic. Every later task builds inside this shape.

Reference direction is the whole point: Domain ← Application ← Adapters ← Host. Nullable + warnings-as-errors on every C# project. Arch rules that police this arrive in Task 4; the structure they police is laid here.

Key design points

PointWhat / why
Anti-god-class Domain Session root guards + routes only; each building block from design/domain-model.md is its own type owning one state slice. Skeleton bodies only (types + minimal state), no phase rules yet. Deterministic class-size gate (Task 4): custom ArchUnitNET rule “no class > 12 methods” fails dotnet test; eslint max-lines/max-statements FE-side. Threshold Ask-first.
Prod sln vs Tests.slnf dotnet build backend/ValuesWorkshop.sln builds zero test projects; dotnet test backend/ValuesWorkshop.Tests.slnf runs all four. Keeps prod build fast + deployable artifact clean.
Per-screen DI contexts Each of facilitator/participant/presenter wraps its tree in its own React context injecting port implementations. Concrete stub port + adapter + context per group so the pattern is code, not a comment.
i18n from stub one Every config JSON field that reaches a screen carries de + en: value names, questions, answers, learning texts, animal names.
OIDC devtool oidc-provider package serving /.well-known/openid-configuration; client/user config deferred to Task 8.

Success criteria (all must pass before PR)

  • pnpm --dir frontend dev|test|lint all run green (Jest: ≥1 real test per layer stub)
  • dotnet build backend/ValuesWorkshop.sln green, no test projects built
  • dotnet test backend/ValuesWorkshop.Tests.slnf runs all 4 test projects green
  • Project references encode dependency direction (no Domain→Adapters edge exists)
  • Each FE screen group has its own DI context; no cross-screen imports
  • curl on the oidc discovery endpoint returns valid JSON

Slices & boundaries

Implementation slices

  1. backend/ — prod projects + sln + slnf + test projects, typed Domain building blocks, green build+test
  2. frontend/ — Next.js init, layer folders, screen groups + DI contexts, green dev/test/lint
  3. config/ + devtools/oidc/ — stubs, discovery endpoint, full acceptance run

Boundaries

  • Always pnpm only · TS strict · nullable + warnings-as-errors · test-first where logic exists
  • Ask first any runtime dep beyond Next.js/Jest/eslint, ASP.NET Core/xUnit, oidc-provider · layout deviations
  • Never feature/phase logic · old repo (CLEANROOM.md) · Session root containing building-block rules
