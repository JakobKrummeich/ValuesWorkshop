Objective

design/screens.md — what each of the three screens (Facilitator · Presenter view · Participant) shows and allows in each of the nine phases, plus low-fi wireframes. Ubiquitous language from domain-model.md; per-phase behavior from state-machine.md (T1–T23). No visual design — colors/typography/spacing come with design tokens in Task 2.

Content

  • 3×9 matrix — 27 cells: per screen per phase, sees / can do (commands issued). Presenter column always "sees only". Cells with nothing new marked intentionally empty.
  • Command traceability table — every 0.1 command (all 22) → screen + wireframe element that issues it; System commands marked as such (no element); implicit commands (JoinSession on QR arrival) marked implicit.
  • Low-fi ASCII wireframes (portrait-mobile-first for participant) — full draft set below.
  • Cross-cutting notes: reconnect/resume view per role · facilitator control bar (AdvancePhase + phase sub-controls, high density, laptop) · presenter fullscreen, zero interactivity, unauthenticated · participant portrait-first, usable on landscape laptop · i18n: every visible string de+en, i18n-relevant states flagged (learning texts, values, group names, labels).

SuccessAll 27 cells filled or intentionally empty · every 0.1 command reachable from a wireframe element (traceability complete) · consistent with SPEC.md screens table and every T1–T23 transition · ubiquitous language only.

BoundariesDoc-only, no code · low-fi only, no visual design · deviations from 0.1/0.2 discovered here → update those docs in same PR (Ask-first) — two already queued, see above.
